v41 · 15 Jun 2026
  • Fresh-breakout structural admission (Commit D). New detectFreshBreakout flags a stock only when all structural criteria hold at once: a long tight base (110-bar window, range <30%), a volume-expansion breakout clearing the base high, price within 5% of the 250-bar high, and EMA20>EMA50 with price>EMA200 and EMA20 rising. Such a stock — a new trend with no backtest edge yet — that the MC gate would otherwise IGNORE is surfaced as WATCH (reason WATCH_FRESH_BREAKOUT), but only if it also clears score≥60 and is reachable. It can never become a BUY, and a PROVEN_NEGATIVE stock is still hard-IGNORE'd (invariant, checked first). Thresholds are a-priori, not tuned. Mirrored in kite.js v4.34.
v40 · 15 Jun 2026
  • Confidence-aware edge gate (Commit C). The binary expectancy<0 → IGNORE is replaced by a 3-state split using edgeClass. PROVEN_NEGATIVE (band entirely below 0) stays a hard IGNORE — the safety invariant, never overridden. A NEGATIVE-but-INDETERMINATE expectancy (band straddles 0 = statistically "unknown", not a proven loser) that otherwise clears score, data and reachability now surfaces as WATCH (reason WATCH_INDETERMINATE) instead of being hard-rejected. Positive-expectancy stocks are untouched, so no BUY is affected; the only new transition is IGNORE→WATCH. Mirrored in kite.js v4.33 (gate now carries edgeClass into the verdict).
v39 · 15 Jun 2026
  • Edge-confidence instrumentation (Commit A — report-only). Two new backtest fields: expSE (standard error of expectancy = std of per-trade R ÷ √n) and edgeClass (PROVEN_POSITIVE / INDETERMINATE / PROVEN_NEGATIVE, from a one-sided confidence band). The backtest line now shows a small edge+/edge?/edge− tag with the SE. This changes no verdict, score, ranking, or gate — it is measurement only, the foundation for the confidence-aware edge gate (Commit C). Mirrored in kite.js v4.31. Differential: BUY/WATCH/IGNORE unchanged, 0/0/0.
v38 · 15 Jun 2026
  • F2 — symmetric edge veto (the "worst stock wins" fix). The Monte-Carlo veto was lopsided: a 0%-MC stock got a small-sample rescue (exp>0 && bt<15) that 1–29%-MC stocks never got, so NAM-INDIA (MC 0%, exp +0.007R) earned BUY while GABRIEL (MC 18.8%, exp +0.154R) — strictly better — was IGNORE'd. Both tiers are now treated identically: any MC<30% is vetoed unless expectancy is genuinely strong (≥0.2R). Differential over 65 audited stocks: exactly one verdict changes — NAM-INDIA BUY→IGNORE. Mirrored byte-identically in kite.js v4.30 so all paths agree.
  • Score alignment (79-vs-87 fix). The Telegram alert & the ≥70 gate used to fire before RS enrichment, so they read the pre-RS score (ANGELONE alerted "79" while its card showed 87). BUY-alert dispatch now runs in a dedicated pass after RS enrichment, so the alert, the ≥70 gate, and the card all use the same RS-adjusted number.
  • Unified alert rule across all paths. Manual scan, "load & analyse", cron, and "run pipeline now" now share one trigger: verdict BUY and score ≥ 70 (RS-adjusted). The cron's old DS/ST/ADX integrity gate was dropped — it was blocking genuine high-conviction BUYs (e.g. ANGELONE, score 87 but ADX 13). kite.js v4.30 ports the RS adjustment so its score matches the browser exactly.

v34 · 14 Jun 2026
  • Manual-scan Telegram + phantom error-card fix (scope bug). maybeSendBuyAlert() was defined inside the Discovery-bot IIFE (const qeDiscovery=(function(){…})), but startScan() lives at the top level, so when a manual scan hit a BUY the call threw "maybeSendBuyAlert is not defined". That ReferenceError was swallowed by the per-stock try/catch, which (a) rendered a phantom "Could not load this stock" error card for the symbol and (b) pushed it into the failed list — so a single symbol counted as 2 scanned, and (c) the throw happened before the Telegram send, so no alert ever went out. All three symptoms (the "2" count, the duplicate error card, the missing notification) were this one bug, live since v23 when startScan was first routed through maybeSendBuyAlert. Fix: the function is hoisted to the top level (right after sendTelegram), so it is reachable from BOTH startScan and the discovery bot; the duplicate copy inside the closure is removed (single canonical definition). Verified by executing the page's scripts in a sandbox: a top-level peer of startScan, and a function in the discovery-bot block, both resolve maybeSendBuyAlert with no ReferenceError. The v32 send-reliability logic (only dedup on confirmed success) is retained.
v33 · 14 Jun 2026

Implemented as one shared predicate (_edgeWatchKind) called from finalDecision, computeExecutionDecision and all three commentary builders, so the verdict pill (Elite execution action), the BUY/WATCH counter and the discovery bot all agree — no counter-vs-pill mismatch. Kill-switch: set QE_EDGE_WATCH=false. No BUY action elements (GTT / Log / full-size) ever fire on these — they remain a watch, not an entry.
v32 · 14 Jun 2026
  • Telegram BUY-alert reliability fix. sendTelegram() previously swallowed every error (network, bad/expired bot token, HTTP 4xx, telegram ok:false) in a bare try/catch and returned nothing, while maybeSendBuyAlert() marked the symbol as "sent" in the per-day dedup map (qe_tg_buysent_<date>) unconditionally after the await. Result: a single silent failure delivered no message AND permanently blocked retries for that symbol for the rest of the day (proven: ABDL BUY score 90 on 14 Jun — verdict/score/creds all valid, BUY≥70 counter = 1 — yet no alert). Now: sendTelegram() checks resp.ok AND the Telegram ok:true body, surfaces failures via a toast + console, and returns a boolean; maybeSendBuyAlert() only writes the dedup entry when the send actually succeeded, so failed sends retry on the next scan instead of being silently poisoned. No change to the score≥70 gate or message format.

v29 · 14 Jun 2026
  • Regime = single source of truth (cross-surface score consistency). The manual scan computed its own Nifty regime (Yahoo) with an intraday ±1.5% override + holiday rule, while cron / discovery bot / QE-gate / Telegram all scored under the server's pure-structural pipeline regime. On a green day the override cancelled a legitimately-earned bear penalty, so the same stock read e.g. 65/WAIT in a manual scan but 53/IGNORE everywhere else (65 × 0.82 = 53.3, the bear multiplier).
